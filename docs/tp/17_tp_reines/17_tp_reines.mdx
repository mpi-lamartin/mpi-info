## Premières questions

1. Donner une solution avec une reine pour $n = 3$ et avec trois reines pour $n = 4$.

<details className="correction-details">
<summary>Correction</summary>

Pour $n = 3$, la reine au centre couvre tout l'échiquier.

Pour $n = 4$, une solution à trois reines est donnée par les positions

```text
(1, 1), (2, 3), (3, 0)
```

soit

```text
. . . .
. x . .
. . . x
x . . .
```

</details>

2. Montrer que le problème admet toujours une solution optimale et majorer simplement sa valeur.

<details className="correction-details">
<summary>Correction</summary>

Pour $n \ge 4$, on sait qu'il existe un placement de $n$ reines sans prises. Comme chaque ligne contient alors une reine, toute case est couverte par la reine de sa ligne. On obtient donc une solution de taille $n$.

Pour $n \in \{1,2,3\}$, on vérifie directement qu'une solution existe. Comme l'ensemble des placements possibles est fini, une solution optimale existe toujours.

On a donc simplement

$$
1 \le r_n \le n.
$$

</details>

3. Proposer un minorant de la valeur optimale.

<details className="correction-details">
<summary>Correction</summary>

Une reine couvre au plus sa ligne, sa colonne et ses deux diagonales, soit au plus $4n - 3$ cases distinctes. Si $r_n$ désigne le nombre minimal de reines, il faut donc au moins couvrir $n^2$ cases avec $r_n$ reines, d'où

$$
r_n \ge \left\lceil \frac{n^2}{4n - 3} \right\rceil.
$$

Ce minorant est simple mais déjà utile pour cadrer l'ordre de grandeur.

</details>

## Recherche exhaustive

4. Implémenter `finale : couverture -> bool`.

<details className="correction-details">
<summary>Correction</summary>

Dans la version finale du fichier, on maintient l'invariant `z = nombre de cases non couvertes`. On peut donc tester la complétude en temps constant.

```ocaml
let finale (c : couverture) : bool =
	c.z = 0
```

</details>

5. Implémenter `mise_a_jour`.

<details className="correction-details">
<summary>Correction</summary>

On incrémente ou décrémente toutes les cases de la ligne, de la colonne et des deux diagonales de la reine. La fonction auxiliaire `modifie_case` maintient aussi `z` quand une case passe de `0` à une valeur positive, ou réciproquement.

```ocaml
let modifie_case (c : couverture) (i : int) (j : int) (v : int) : unit =
	let avant = c.m.(i).(j) in
	let apres = avant + v in
	assert (apres >= 0);
	if avant = 0 && apres > 0 then c.z <- c.z - 1
	else if avant > 0 && apres = 0 then c.z <- c.z + 1;
	c.m.(i).(j) <- apres

let mise_a_jour (c : couverture) (i : int) (j : int) (v : int) : unit =
	assert (c.m.(i).(j) = 0 && v = 1 || c.m.(i).(j) = 1 && v = -1);
	let n = Array.length c.m in
	for colonne = 0 to n - 1 do
		modifie_case c i colonne v
	done;
	for ligne = 0 to n - 1 do
		if ligne <> i then modifie_case c ligne j v
	done;
	(* puis les quatre demi-diagonales *)
	...
```

L'idée clé est qu'une case contenant une reine vaut nécessairement `1`, car le placement est sans prises.

</details>

6. Rappeler le principe du branch and bound.

<details className="correction-details">
<summary>Correction</summary>

On explore récursivement un arbre de décisions. À chaque nœud, on compare la meilleure solution déjà connue avec ce qu'il est encore possible d'obtenir dans la branche courante. Si cette branche ne peut plus battre le meilleur optimum courant, on l'élague immédiatement.

</details>

7. Décommenter `couverture_par_reines` et corriger ses erreurs.

<details className="correction-details">
<summary>Correction</summary>

Les principales corrections sont :

- utiliser des références pour `r_min` ;
- créer correctement la couverture initiale ;
- ajouter les `do`, `then` et `;` manquants ;
- conserver l'invariant de cohérence de `c` après chaque retour récursif.

```ocaml
let couverture_par_reines n =
	let r_min = ref n in
	let c = creer_couverture n in
	let rec place i r =
		if r >= !r_min then ()
		else if finale c then r_min := r
		else if i < n then begin
			place (i + 1) r;
			for j = 0 to n - 1 do
				if c.m.(i).(j) = 0 then begin
					mise_a_jour c i j 1;
					place (i + 1) (r + 1);
					mise_a_jour c i j (-1)
				end
			done
		end
	in
	place 0 0;
	!r_min
```

</details>

9. Proposer un élagage simple.

<details className="correction-details">
<summary>Correction</summary>

Le plus simple consiste à arrêter immédiatement l'exploration dès que l'on a déjà placé au moins autant de reines que la meilleure solution connue :

```ocaml
if r >= !r_min then ()
```

Cet élagage est très rentable car il coupe toutes les branches qui ne peuvent plus améliorer l'optimum courant.

</details>

10. Modifier la fonction pour renvoyer aussi une solution optimale.

<details className="correction-details">
<summary>Correction</summary>

Il suffit de mémoriser la liste courante des placements et de la copier quand on améliore l'optimum.

```ocaml
let couverture_par_reines_positions n =
	let r_min = ref n in
	let meilleure_solution = ref [] in
	let c = creer_couverture n in
	let rec place i r positions =
		if r >= !r_min then ()
		else if finale c then begin
			r_min := r;
			meilleure_solution := List.rev positions
		end
		else if i < n then begin
			place (i + 1) r positions;
			for j = 0 to n - 1 do
				if c.m.(i).(j) = 0 then begin
					mise_a_jour c i j 1;
					place (i + 1) (r + 1) ((i, j) :: positions);
					mise_a_jour c i j (-1)
				end
			done
		end
	in
	place 0 0 [];
	(!r_min, !meilleure_solution)
```

</details>

12. Maintenir l'invariant `z` et obtenir `finale` en $O(1)$.

<details className="correction-details">
<summary>Correction</summary>

On initialise `z` à `n * n`. Ensuite, chaque appel à `modifie_case` met `z` à jour :

- `z := z - 1` quand une case passe de `0` à une valeur positive ;
- `z := z + 1` quand une case repasse de positive à `0`.

On obtient alors `finale c = (c.z = 0)`, donc en temps constant.

</details>

13. Compter le nombre de solutions optimales.

<details className="correction-details">
<summary>Correction</summary>

On garde un compteur séparé.

- quand on trouve une solution strictement meilleure, on remplace `r_min` et on remet le compteur à `1` ;
- quand on retrouve une solution de même taille optimale, on incrémente ce compteur.

```ocaml
let couverture_par_reines_nombre_solutions n =
	let r_min = ref n in
	let nb_solutions = ref 0 in
	let c = creer_couverture n in
	let rec place i r =
		if r > !r_min then ()
		else if finale c then begin
			if r < !r_min then begin
				r_min := r;
				nb_solutions := 1
			end
			else incr nb_solutions
		end
		else if i < n then begin
			place (i + 1) r;
			for j = 0 to n - 1 do
				if c.m.(i).(j) = 0 then begin
					mise_a_jour c i j 1;
					place (i + 1) (r + 1);
					mise_a_jour c i j (-1)
				end
			done
		end
	in
	place 0 0;
	(!r_min, !nb_solutions)
```

</details>

## Approche probabiliste

15. Différence entre Monte-Carlo et Las Vegas.

<details className="correction-details">
<summary>Correction</summary>

Un algorithme de Monte-Carlo a un temps d'exécution borné, mais peut produire un résultat faux avec une certaine probabilité. Un algorithme de Las Vegas produit toujours un résultat correct, mais sa durée d'exécution est aléatoire.

</details>

16. Déterminer la probabilité `p_i` du procédé de tirage uniforme en une passe.

<details className="correction-details">
<summary>Correction</summary>

Il faut prendre

$$
p_i = \frac{1}{i + 1}.
$$

En effet, la probabilité qu'un élément déjà vu reste mémorisé après l'étape $i$ vaut

$$
\left(\frac{1}{i}\right)\left(1 - \frac{1}{i + 1}\right) = \frac{1}{i + 1},
$$

ce qui conserve l'uniformité par récurrence.

</details>

17. Adapter ce procédé à un prédicat `P`.

<details className="correction-details">
<summary>Correction</summary>

On ne compte que les éléments qui vérifient `P`. Si le $k$-ième élément admissible apparaît, on le retient avec probabilité $1/k$. C'est exactement le principe du reservoir sampling sur la sous-suite filtrée.

</details>

18. Implémenter `choix_position`.

<details className="correction-details">
<summary>Correction</summary>

```ocaml
let choix_position (a : int array) : int =
	let resultat = ref (-1) in
	let nb_zeros = ref 0 in
	Array.iteri
		(fun i valeur ->
			if valeur = 0 then begin
				incr nb_zeros;
				if Random.int !nb_zeros = 0 then resultat := i
			end)
		a;
	!resultat
```

Chaque indice nul est bien choisi avec la même probabilité, et le tableau n'est parcouru qu'une seule fois.

</details>

19. Implémenter `couverture_par_reines_aleatoire`.

<details className="correction-details">
<summary>Correction</summary>

On répète `nb_essais` tentatives indépendantes. À chaque ligne, avec probabilité $(n-1)/n$, on place éventuellement une reine sur une case encore nulle choisie uniformément.

```ocaml
let couverture_par_reines_aleatoire n nb_essais =
	Random.self_init ();
	let meilleur = ref n in
	for _ = 1 to nb_essais do
		let c = creer_couverture n in
		let nb_reines = ref 0 in
		let abandon = ref false in
		for i = 0 to n - 1 do
			if not !abandon then begin
				if !nb_reines >= !meilleur then abandon := true
				else if Random.int n < n - 1 then begin
					let j = choix_position c.m.(i) in
					if j <> -1 then begin
						mise_a_jour c i j 1;
						incr nb_reines
					end
				end
			end
		done;
		if not !abandon && finale c && !nb_reines < !meilleur then
			meilleur := !nb_reines
	done;
	!meilleur
```

</details>

20. Donner un majorant expérimental pour $n \in \{20,30,100\}$ avec $10^6$ essais.

<details className="correction-details">
<summary>Correction</summary>

Il n'y a pas ici de réponse unique : toute exécution correcte fournit un majorant valable. En pratique, il faut compiler en natif avec `ocamlopt` pour que $10^6$ essais soient réalistes, et les valeurs obtenues dépendent de la graine aléatoire.

</details>

21. Complexité de cette approche.

<details className="correction-details">
<summary>Correction</summary>

Un essai parcourt les $n$ lignes. À chaque ligne, le choix et la mise à jour coûtent au plus $O(n)$. On obtient donc une complexité totale en

$$
O(nb\_essais \times n^2).
$$

</details>

22. Proposer un autre choix de `p`.

<details className="correction-details">
<summary>Correction</summary>

Une heuristique naturelle consiste à adapter `p` à la marge encore disponible avant de dépasser la meilleure solution connue. Par exemple, à la ligne `i`, si l'on a déjà placé `r` reines et que le meilleur majorant vaut `m`, on peut prendre une probabilité croissante avec

$$
\frac{m - r}{n - i}.
$$

Cela pousse l'algorithme à poser davantage de reines quand il en manque encore beaucoup, et à devenir plus parcimonieux lorsqu'il approche du meilleur score courant.

</details>

23. Commenter les proportions d'échecs, d'élagages et d'améliorations.

<details className="correction-details">
<summary>Correction</summary>

En général :

- les échecs restent majoritaires ;
- les élagages deviennent de plus en plus fréquents dès qu'un bon majorant a été trouvé ;
- les améliorations deviennent rapidement rares.

C'est exactement le comportement attendu d'une heuristique aléatoire avec branch and bound.

</details>

## Parallélisation

24. Proposer une version parallélisée.

<details className="correction-details">
<summary>Correction</summary>

Le plus simple est de répartir les essais aléatoires entre plusieurs fils, chacun effectuant un bloc d'essais indépendant, puis de protéger la meilleure valeur globale par un mutex.

```ocaml
let couverture_par_reines_aleatoire_parallel n nb_essais nb_threads =
	let meilleur = ref n in
	let verrou = Mutex.create () in
	let essais_par_thread = nb_essais / nb_threads in
	let travail () =
		let local = couverture_par_reines_aleatoire n essais_par_thread in
		Mutex.lock verrou;
		if local < !meilleur then meilleur := local;
		Mutex.unlock verrou
	in
	let fils = Array.init nb_threads (fun _ -> Thread.create travail ()) in
	Array.iter Thread.join fils;
	!meilleur
```

</details>

27. Déterminer le minimum pour $n = 4$ sans imposer le placement sans prises.

<details className="correction-details">
<summary>Correction</summary>

On peut couvrir l'échiquier $4 \times 4$ avec deux reines, par exemple en $(0, 0)$ et $(2, 2)$. Une seule reine ne suffit pas sur $4 \times 4$, donc le minimum vaut `2`.
